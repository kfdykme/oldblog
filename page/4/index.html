Chapter3 处理机调度与死锁

3.2 调度队列模型和调度准则

三种调度队列模型

  1. 仅有进程调度
  2. 具有高级和低级调度

  3. 具有三级调度

    3.2.1 调度队列模型

  4. 仅有进程调度的调度队列模型(分时系统)

    • 分时系统中,通常仅设有进程调度
    • 系统把这些进程组织成一个就绪队列
    • 每个进程在执行时,可能有以下几种情况
      • 进程获得cpu正在执行
      • 任务在给定的时间片内完成,释放处理机后为完成状态
      • 任务在给定时间片内未完成,进入就绪队列末尾
      • 在执行期间因某事件而阻塞
    • 就绪队列的时间片轮转
      • 常采用FCFS(FIFO)算法,FCFS(FIFO)队列
    • 进程执行时的三种情况
      • 完成,时间片到,阻塞

  5. 具有高级和低级调度的调度队列模型

    • 在批处理系统中,不仅需要进程调度,而且还要有作业调度
    • 就绪队列的形式
      • 在批处理系统中,常用高优先权队列.进程进入就绪队列,按优先权高低插入相应位置,调度程序总是把处理机分配给就绪队列首进程
    • 设置多个阻塞队列
      • 根据事件的不同设置多个阻塞队列提高效率
    • 常采用高优先权优先调度算法
      • 可采用优先队列,进程来时按优先权插队,从队首调度(效率高)
      • 可采用无序队列,每次调度时,先比较优先度
    • 多个阻塞队列

  6. 同时具有三级调度的调度队列模型

    • 在Os中引入中级调度后,进程的就绪状态分为内存就绪(表示进程在内存中就绪)和外存就绪(在外存中就绪)
    • 同时,阻塞状态分为内存阻塞和外存阻塞
    • 调出操作
      • 内存就绪-> 外存就绪
      • 内存阻塞-> 外存阻塞
    • 中级调度
      • 外存就绪 -> 内存就绪

3.2.2 选择调度方式和调度算法的若干准则

  • 在不同系统中通常采用不同的调度方式和算法
  • 调度的目标
    1. 提高处理机的利用率
    2. 提高系统的吞吐量
    3. 尽量减少进程的响应时间
    4. 防止进程长期得不到运行

  • 系统选择调度方式和算法的准则分为两种

    • 面向用户的准则
    • 面向系统的准则

  • 面向用户的准则

    1. 周转时间短(评价批处理)
      1. 作业在外存后备队列上等待调度的时间
      2. 进程在就绪队列等待调度的时间
      3. 进程在cpu执行的时间
      4. 等待i/o操作完成的时间
        • 带权周转时间 : 作业的周转时间T与系统为它提供服务的时间T_s之比,即W=T/T_s
        • 用来评价批处理系统的性能、选择作业调度方式与算法的重要准则之一

    2. 响应时间快(评价时分)

      1. 从键盘输入的请求信息传送到处理机的时间
      2. 处理机对请求信息进行处理的时间
      3. 将所形成的响应时间回送到终端显示器的时间
        • 响应时间 : 从用户通过键盘提交一个请求开始,知道系统首次产生响应所需要的时间
        • 用来评价分时系统的性能、选择进程调度算法的重要准则之一

    3. 截止时间的保证(评价实时)

      1. 开始截止时间
      2. 终止/完成截止时间
        • 截止时间 : 某任务必须开始执行的最迟时间,或必须完成的最迟时间.(deadline)
        • 用来评价实时系统的性能、选择实时调度算法的重要准则之一
    4. 优先权准则
      • 让某些紧急的作业能得到及时处理
      • 往往还需选择抢占调度方式,才能保证紧急作业得到及时处理
      • 适合批处理,分时,实时系统

  • 面向系统的准则

    1. 系统吞吐量高(评价批处理系统)
      • 吞吐量
      • 与批处理作业的平均长度有关
    2. 处理机利用率高.
      • 主要对大中型用户系统,对单用户和实时系统不重要
      • cpu利用率 = cpu有效工作时间/(cpu有效工作时间+ cpu空闲等待时间)
    3. 各类资源的平衡利用.(内存,外存,i/o设备等)
      • 主要对大中型系统,对微型机和实时系统不重要

  • 引起进程调度的因素可归结为

    1. 正在执行的进程执行完毕,或因发生某事件而不能再继续执行(包括:当前执行进程被中断,时间片用完了,挂起自己,推出等)
    2. 执行中的进程因提出i/o请求而暂停执行
    3. 在进程通信或同步过程中执行了某种原语操作,如p,v操作原语,block原语,wakeup原语等

  • 进程切换

    • 当一个进程占用处理机执行完(或不能继续执行),则换另一个进程占用处理机执行
    • 在进行时要保护进程的执行现场(进程的上下文)
  • 进程调度
    • 把处理机分配给不同的进程占用执行

  • 调度程序

    • 实现分配处理机的程序

  • 进程切换的基本步骤

    1. 保存进程上下文环境
    2. 更新当前运行进程的控制块内容,将其状态改为就绪或阻塞状态
    3. 将进程控制块移到相应队列(就绪队列或阻塞队列)
    4. 改变需投入运行进程的进程控制块内容,将其内容状态改变为运行状态
    5. 恢复需投入运行进程的上下文环境

  • linux的进程切换(context_switch())

    1. 进程页表PGD 切换
    2. 内核堆栈和硬件上下文切换(包含CPU寄存器)

3.1 处理机调度的层次

发表于 2018-05-08

Chapter3 处理机调度与死锁

3.1 处理机调度的层次

  • 在多道程序系统中,一个作业从提交到执行,通常都要经历多层调度,如
    • 高级调度 决定哪些程序可以进入系统
    • 中级调度 决定内存中程序的位置和状态
    • 低级调度 决定cpu内存中程序的位置和状态
    • i/o调度
  • 系统的运行性能在很大程度上取决于调度
    • 如吞吐量的大小,周转时间的长短,响应的及时性等
  • 调度是多道系统的关键

3.1.1 高级调度

  • 也称作业调度,长程调度或接纳调度

  • 主要功能

    • 根据某种算法,把外存中处于后备队列中的那些作业调入内存

  • 批处理系统需要有作业调度,分时和实时系统无需此调度

  • 每次执行作业调度时
    1. 接纳多少各作业(取决于多道程序度)
      • 多->服务质量下降
      • 少->资源利用率低
    2. 接纳哪些作业 (取决于采用的调度算法)

  • 名词:

    • 多道程序度 : 即允许多少个作业同时在内存中运行
    • 周转时间 : 从作业被提交给系统起,到作业完成为止的这段时间间隔
    • 吞吐量 : 是指在单位时间内系统所完成的作业数
    • 响应比 = (等待时间+要求服务时间)/要求服务时间 = 响应时间/要求服务时间

  • 主要用于批处理系统,其设计目标时最大限度地发挥各种资源地利用率和保持系统内各种活动地充分并行

3.1.2 低级调度

  • 又称进程调度或短程调度,它所调度地对象时进程,三种OS都必须配置这级调度(最基本调度)
  • 用于决定就绪队列中的哪个进程应获得处理机,然后再由分派进程执行把处理机分配给该进程的具体操作
  • 进程调度方式
    1. 非抢占方式 进程占用处理机直至自愿放弃或发生某事件被阻塞时,在把处理机分配给其他进程
    2. 抢占方式 允许暂停某个正在执行的进程,将处理机重新分配给另一个进程
  • 3个基本机制
    1. 排队器 为了提高进程调度的效率,应事先将系统中的所有就绪的进程按照一定的方式排成一个或多个队列
    2. 分派器(调度程序) 分派器把由进程调度程序所选定的进程从就绪队列中取出,然后进行上下文切换,将处理机分配给他
    3. 上下文切换机制 当对处理机进行切换时,会发生两对上下文切换操作
  • 功能
    • 按某种算法选区进程(调度)
    • 保存处理机的现场信息(上下文切换sept 1)
    • 把处理器分配给进程(上下文切换step 2)

  • 时间尺度 通常是毫秒级的
    -

  • 分配程序的主要功能

    • 进行进程切换
    • 转到用户态
    • 开始执行被选中的进程

  • 非抢占方式

    • 引起调度的因素
      • 正在执行的进程执行完毕,或者因发生某事件而不能再继续执行
      • 执行中的进程因提出i/o操作而暂停执行
      • 在进程通信或同步过程中执行了某种原语操作,如wait,block,wakeup原语
    • 优点
      • 算法简单,系统开销小
    • 缺点
      • 紧急任务不能及时响应
      • 短进程要等到长进程允许结束

  • 抢占方式

    • 优点
      • 可以防止一个长进程长时间占用处理机
      • 能为大多数进程提供更为公平的服务
      • 能满足对响应时间有着较严格要求的实时任务的需求
    • 缺点
      • 抢占方式比非抢占方式调度所需的开销较大
      • 调度算法复杂
    • 抢占的原则
      1. 时间片原则
        • 适用于分时,大多数实时以及要求较高的批处理系统
      2. 优先权原则
        • 重要紧急任务优先权高
      3. 短作业(进程)优先原则

3.1.3 中级调度

  • 又称中程调度
  • 主要目的
    • 提高内存的利用率和系统的吞吐量
  • 具体实现
    • 使那些暂时不能运行的进程不再占用宝贵的内存资源,而将其调至外存去等待,把此时的进程状态称为就绪驻外存状态或挂起状态.当这些进程又具备运行条件,且内存又稍有空闲时,由中级调度来决定把外存上的那些又具备运行条件的就绪进程,重新调入内存,并修改其状态为就绪状态,挂在就绪队列上等待进程调度

处理机调度的层次

  • 终端型作业：低级。
  • 批量性作业：高级—低级。
  • 现代较完善的os具有三级调度。

end

3.8 死锁的检测与解除

  • 如果系统不愿意附加太多约束条件预防死锁,也不希望系统额外开销预测并避免死锁,那么只能允许死锁出现,然后解除它
  • 因此,需要有某种方法来检测死锁

3.8.1 死锁的检测

  1. 资源分配图
  2. 死锁定理 : S为死锁当且仅当S状态的资源分配图是不可完全简化的
  3. 死锁检测中的数据结构
    1. 可利用资源向量Available ,他表示m类资源中每一类资源的可用数目
    2. 把不占用资源的进程(Available = 0)记入L表中,即L_i ∪L
    3. 从进程集合中找到一个Request_i < work的进程,作如下处理
      1. 将其资源分配图简化,释放出资源,增加工作向量Work=Work+ Available
      2. 将他记入L表
    4. 若不能把所有进程都记入L表,便表明系统状态s的资源分配图是不可完全简化的.因此,该系统即将发生死锁

3.8.2 死锁的解除

  • 常用两种解除死锁方法

    1. 剥夺资源,从其他进程剥夺足够数量的资源给死锁进程,以解除死锁状态
    2. 撤销进程,最简单的撤销进程的方法,是使全部死锁进程都夭折掉;或按某种顺序逐渐逐个撤销进程,直至有足够的资源可用,使死锁状态消除为止

  • 按照解除死锁复杂度递增的顺序列出解除死锁的方法

    1. 撤销死锁进程
    2. 把死锁进程恢复到前一个检查点,重新执行每个进程
    3. 按某种原则逐个选择死锁进程进行撤销,直到解除系统死锁
    4. 按照某种原则逐个剥夺进程资源,直到解除死锁

  • 最小代价原则

    • 上述第三种第四种方法需要选择系统付出代价最少的进程,最小代价原则
      • 到目前为止，花费处理机的时间最少的进程；
      • 到目前为止，产生输出最少的进程；
      • 估计未执行部分最多的进程；
      • 到目前为止，已获得资源量最少的进程；
      • 优先级最低的进程。

3.7 避免死锁

3.7.1 安全状态

  • 安全状态，是指系统能按某种进程顺序(P1, P2, …，Pn)(称〈P1, P2, …, Pn〉序列为安全序列)，来为每个进程Pi分配其所需资源，直至满足每个进程对资源的最大需求，使每个进程都可顺利地完成。如果系统无法找到这样一个安全序列，则称系统处于不安全状态。

-

3.7.2 利用银行家算法避免死锁

问题描述：
该算法可用于银行发放一笔贷款前，预测该笔贷款是否会引起银行资金周转问题。
这里，银行的资金就类似于计算机系统的资源，贷款业务类似于计算机的资源分配。银行家算法能预测一笔贷款业务对银行是否是安全的，该算法也能预测一次资源分配对计算机系统是否是安全的。
为实现银行家算法，系统中必须设置若干数据结构。

银行家算法思想——死锁避免策略：
1.当前状态下，某进程申请资源；
2.系统假设将资源分给该进程，满足它的需求；
3.检查分配后的系统状态是否是安全的，如果是安全，就确认本次分配；如果系统是不安全的，就取消本次分配并阻塞该进程。（第三步又称安全算法）
避免死锁策略也称作银行家算法。

3.5 产生死锁的原因和必要条件

发表于 2018-05-07

3.5 产生死锁的原因和必要条件

3.5.1 产生死锁的原因

  1. 资源竞争
  2. 进程间推进顺序不当
  1. 竞争不可抢占性资源引起死锁
  2. 经侦临时性(消耗性)资源引起死锁
  3. 进程推进顺序不当引起死锁
  • 竞争资源,未必产生死锁
  • 是否产生死锁,还取决于动态执行和应用程序细节

3.5.2 产生死锁的必要条件

  1. 互斥条件 : 进程对分配到的资源进行排他性使用
  2. 请求和保持条件
  3. 不剥夺条件
  4. 环路等待条件

3.5.3 处理死锁的基本方法

  1. 预防死锁
    • 设置某些限制条件,破坏四个必要条件中的一个或几个
    • 优点:
      • 容易实现
    • 缺点:
      • 资源利用率低或吞吐量降低
  2. 避免死锁
    • 再资源动态分配过程中使用某种方法防止系统进入不安全状态
    • 优点
      • 较弱限制条件可获得较高系统资源利用率和吞吐量
    • 缺点
      • 有一定实现难度
  3. 检测死锁
    • 预先不采取任何限制,也不检查是否已进入不安全区,通过设置检测机构,检查出死锁后解除
  4. 解除死锁
    • 常用撤销或挂起一些进程,回收一些资源
